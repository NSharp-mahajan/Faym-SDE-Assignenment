# Database Design

## Overview
The system relies on MongoDB, a NoSQL database, structured using Mongoose schemas to enforce data types, validations, and constraints. The design mimics a relational ledger system to ensure financial consistency.

---

## Collections

### 1. User
**Purpose**: Stores affiliate user identities and basic information. Acts as the root entity for all relational queries (sales, payouts, withdrawals).

| Field | Data Type | Constraints | Description |
|-------|-----------|-------------|-------------|
| `_id` | ObjectId | Auto-generated | Unique identifier. |
| `name` | String | Required, Trimmed | The user's full name. |
| `email` | String | Required, Unique, Lowercase, Trimmed | Contact and login email. |
| `createdAt` | Date | Auto-generated | Account creation timestamp. |
| `updatedAt` | Date | Auto-generated | Last modification timestamp. |

---

### 2. Sale
**Purpose**: Records individual conversion events generated by the affiliate, tracking their earnings and approval lifecycle.

| Field | Data Type | Constraints | Description |
|-------|-----------|-------------|-------------|
| `_id` | ObjectId | Auto-generated | Unique identifier. |
| `userId` | ObjectId | Ref: User, Required, Indexed | The affiliate who generated the sale. |
| `earnings` | Number | Required, Min: 0 | Gross commission generated by the sale. |
| `status` | String | Enum: [Pending, Approved, Rejected], Default: Pending | Lifecycle state of the sale. Indexed. |
| `advancePaid` | Boolean | Default: false | Flag indicating if a 10% advance was processed. |
| `advanceAmount`| Number | Default: 0 | Exact amount of the advance paid. |
| `reconciledAt` | Date | Default: null | Timestamp when status transitioned to Approved/Rejected. |

---

### 3. Payout
**Purpose**: Acts as an immutable ledger for inward flow of funds towards the affiliate. Payouts are generated by the system (Advances, Finals, Adjustments).

| Field | Data Type | Constraints | Description |
|-------|-----------|-------------|-------------|
| `_id` | ObjectId | Auto-generated | Unique identifier. |
| `userId` | ObjectId | Ref: User, Required, Indexed | Affiliate receiving the payout. |
| `saleId` | ObjectId | Ref: Sale, Required, Indexed | The specific sale this payout is linked to. |
| `type` | String | Enum: [ADVANCE, FINAL, ADJUSTMENT], Required | Categorization of the payout logic. |
| `amount` | Number | Required | Value transferred (can be negative for adjustments). |
| `status` | String | Enum: [PENDING, SUCCESS, FAILED], Default: PENDING | Transaction state. |
| `processedAt` | Date | Default: null | Timestamp of successful transaction completion. |

---

### 4. Withdrawal
**Purpose**: Acts as an immutable ledger for the outward flow of funds. Withdrawals are generated when an affiliate cashes out their available balance.

| Field | Data Type | Constraints | Description |
|-------|-----------|-------------|-------------|
| `_id` | ObjectId | Auto-generated | Unique identifier. |
| `userId` | ObjectId | Ref: User, Required, Indexed | Affiliate making the withdrawal. |
| `amount` | Number | Required, Min: 0 | Amount successfully withdrawn. |
| `status` | String | Enum: [PENDING, SUCCESS, FAILED], Default: PENDING | Transaction state. |
| `requestedAt` | Date | Default: Date.now(), Indexed | Time the withdrawal was initiated (used for 24h limit check). |
| `processedAt` | Date | Default: null | Time the funds left the system. |

---

## Entity Relationships

* **User ➔ Sales (1:N)**: A single affiliate User can generate multiple Sales. The `Sale` collection stores the `userId` to establish this relationship.
* **User ➔ Payouts (1:N)**: A User receives multiple Payouts. The system calculates user balance by summing all Payouts referencing their `userId`.
* **User ➔ Withdrawals (1:N)**: A User can request multiple Withdrawals over time. The system calculates depleted funds by summing Withdrawals mapped to their `userId`.
* **Sale ➔ Payouts (1:N)**: A single Sale typically generates two Payout records: an `ADVANCE` payout upon inception, and a `FINAL` or `ADJUSTMENT` payout upon reconciliation. The `Payout` collection tracks the originating `saleId`.
